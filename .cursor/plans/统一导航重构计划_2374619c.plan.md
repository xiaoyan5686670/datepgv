# 统一导航与路由状态改造计划

## 目标
- 统一 `home/users/admin/settings` 的导航结构与视觉风格，避免多处重复 header。
- 建立登录后共享壳层，支持桌面端顶部导航与移动端底部 Tab。
- 将页面内部 tab/section 状态绑定到 URL，刷新、分享链接、前进后退均可恢复。

## 现状结论
- 当前仅有根布局 [`/Users/qinxiaoyan/work/datepgv/frontend/src/app/layout.tsx`](/Users/qinxiaoyan/work/datepgv/frontend/src/app/layout.tsx)，没有“登录后页面”共享布局。
- [`/Users/qinxiaoyan/work/datepgv/frontend/src/app/page.tsx`](/Users/qinxiaoyan/work/datepgv/frontend/src/app/page.tsx)、[`/Users/qinxiaoyan/work/datepgv/frontend/src/app/users/page.tsx`](/Users/qinxiaoyan/work/datepgv/frontend/src/app/users/page.tsx)、[`/Users/qinxiaoyan/work/datepgv/frontend/src/app/admin/page.tsx`](/Users/qinxiaoyan/work/datepgv/frontend/src/app/admin/page.tsx)、[`/Users/qinxiaoyan/work/datepgv/frontend/src/app/settings/page.tsx`](/Users/qinxiaoyan/work/datepgv/frontend/src/app/settings/page.tsx) 各自手写导航。
- `admin/users/settings` 的内部切换状态主要由本地 state 管理，URL 同步不完整。

## 实施方案

### 1) 路由分组重构（不改外部 URL）
- 在 [`/Users/qinxiaoyan/work/datepgv/frontend/src/app`](/Users/qinxiaoyan/work/datepgv/frontend/src/app) 下引入 route group（如 `(authed)`）。
- 将 `page.tsx`、`users/page.tsx`、`admin/page.tsx`、`settings/page.tsx` 迁移到该分组下，保持路径仍为 `/` `/users` `/admin` `/settings`。
- 新增 `(authed)/layout.tsx` 作为共享壳层，承接通用导航与鉴权入口；`login`、`sso/callback` 保持独立。

### 2) 提取统一导航组件
- 新增导航组件目录（建议 `frontend/src/components/navigation/`）。
- 组件拆分建议：
  - `AppTopNav`：桌面顶栏（主导航入口、主题切换、用户信息、可扩展 actions）。
  - `MobileBottomNav`：移动端底部主导航（首页/用户/管理/设置）。
  - `PageSectionTabs`：页面内二级切换（admin section、users tab、settings tab）。
- 页面仅传入“当前 active 项 + 可选右侧操作”，移除重复 HTML。

### 3) URL 状态统一策略
- `admin`：统一 `?section=tables|relations|rag|usage|audit`，点击 tab 时写回 URL。
- `users`：统一 `?tab=list|org`。
- `settings`：统一 `?tab=llm|embedding|analytics|scope`。
- 读取策略：首次渲染从 `searchParams` 派生，非法值回退默认项；切换时使用 `router.replace` 保持无堆叠噪音。

### 4) 面包屑与层级感（轻量首版）
- 在共享顶栏中提供可选 `breadcrumbs` 插槽。
- 首版先覆盖 `admin` 的 section 显示（例如：管理 > 表目录），再扩展其他页面。

### 5) 迁移与回归顺序
- 第一批：`admin/users/settings` 接入共享壳层与统一导航。
- 第二批：首页接入共享壳层，保留其会话侧栏能力。
- 第三批：补齐 breadcrumb 显示与 URL 参数校验边界。
- 每步都做桌面/移动端回归，确保权限可见性逻辑（admin 与非 admin）不变。

## 验收标准
- 代码层：导航结构集中到共享 layout + navigation 组件，页面不再重复 header 框架。
- 体验层：移动端所有核心页面均可通过底部 Tab 到达，不再依赖“返回按钮模式”。
- 状态层：`admin/users/settings` 在刷新与分享链接后可恢复到对应 tab/section。
- 一致性：主题切换、用户信息、主导航入口在四个页面表现一致。

## 风险与缓解
- 风险：route group 迁移引发导入路径/相对样式引用变化。
  - 缓解：分批迁移并先保留旧组件适配层，完成后再清理。
- 风险：URL 状态与本地 state 双源冲突。
  - 缓解：改为“URL 单一事实来源”，本地只保留派生状态。
- 风险：移动端底部 Tab 与页面已有浮动按钮冲突。
  - 缓解：统一内容区底部安全留白（padding-bottom）规范。
