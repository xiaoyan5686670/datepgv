# 最终架构决策指南：Apache AGE vs 独立 Neo4j

在完成 POC（概念验证）和压测后，请根据以下维度进行打分和决策。

## 1. 压测指标对比

| 指标 | 现有 PG (Recursive CTE) | PostgreSQL + Apache AGE | 独立 Neo4j (单体直连) |
| :--- | :--- | :--- | :--- |
| **查询耗时 (1-3层)** | 较高 | 低 | 极低 |
| **查询耗时 (5层以上)** | 极高 (可能超时) | 中等 | 低 |
| **写入延迟** | 极低 (原生事务) | 极低 (原生事务) | 较高 (异步双写延迟) |

## 2. 决策树 (Decision Tree)

### 场景 A：选择 PostgreSQL + Apache AGE (强烈推荐)
如果您的压测结果满足以下条件，请毫不犹豫地选择 AGE：
1. **性能达标**：核心复杂查询（如社交网络、组织架构、推荐链路）在 AGE 中的耗时满足业务 SLA（例如 < 100ms）。
2. **数据量可控**：图节点和边的数量在千万级别或以下。
3. **运维资源紧张**：团队没有专职 DBA，不希望引入新的数据库组件和复杂的数据同步中间件。
4. **强一致性要求高**：业务需要图关系数据和核心业务数据保持绝对的事务一致性。

### 场景 B：选择 独立 Neo4j (单体直连)
如果您的压测结果出现以下情况，请退而求其次选择 Neo4j：
1. **深度遍历性能瓶颈**：业务需要经常进行 5 层以上的深度关系遍历，AGE 的底层表扫描机制导致查询超时。
2. **海量图数据**：图关系数据量达到十亿级别，需要专门的图数据库底层结构（Index-free adjacency）来支撑。
3. **需要高级图算法**：业务需要运行 PageRank、社区发现（Community Detection）、最短路径等复杂的图算法，AGE 暂不支持，而 Neo4j 的 GDS (Graph Data Science) 库提供了原生支持。

## 3. 实施建议
* **如果选择了 AGE**：无需修改基础设施架构。在现有单体工程中，引入支持 Cypher 的 ORM 或直接使用 JDBC/psycopg2 执行 Cypher 语句。
* **如果选择了 Neo4j**：在单体工程中引入 Neo4j Driver。在业务代码中实现**应用层异步双写**（主流程写 PG，异步任务写 Neo4j），并做好重试和补偿机制。**不要新开微服务工程**。
